# Content Guidelines: anti-AI-slop, content rules, scale specs

The traps AI design falls into most easily. This is a "what not to do" list, more important than the "what to do" list – because AI slop is the default, and unless you actively avoid it, it will happen.

## Full AI-slop blacklist

### Visual traps

**❌ Aggressive gradient backgrounds**
- Purple -> pink -> blue full-screen gradient (the signature flavor of AI-generated web pages)
- Rainbow gradients in any direction
- Mesh gradients covering the background
- ✅ If you must use gradient: subtle, monochromatic, applied with intent (e.g. on button hover)

**❌ Rounded cards + left-border accent color**
```css
/* This is the signature of an AI-flavored card */
.card {
  border-radius: 12px;
  border-left: 4px solid #3b82f6;
  padding: 16px;
}
```
This card flavor is everywhere in AI-generated dashboards. Want emphasis? Use a more design-forward approach: background contrast, weight/size contrast, plain dividers, or simply no card at all.

**❌ Emoji decoration**
Unless the brand itself uses emoji (e.g. Notion, Slack), do not put emoji on UI. **Especially do not**:
- Title prefixes like 🚀 ⚡️ ✨ 🎯 💡
- ✅ in feature lists
- → in a CTA button (a standalone arrow is fine; emoji-style arrows are not)

If you have no icons, use a real icon library (Lucide / Heroicons / Phosphor), or use a placeholder.

**❌ Using SVG for imagery**
Do not try to draw with SVG: people, scenes, devices, objects, abstract art. SVG imagery generated by AI screams AI – childish and cheap. **A grey rectangle with a "Illustration slot 1200x800" label beats a clumsy SVG hero illustration by a factor of 100.**

The only places SVG is acceptable:
- Real icons (16x16 to 32x32)
- Geometric shapes as decorative elements
- Data viz charts

**❌ Iconography overload**
Not every title / feature / section needs an icon. Overusing icons makes the interface feel like a toy. Less is more.

**❌ "Data slop"**
Made-up decorative stats:
- "10,000+ happy customers" (you have no idea if there are)
- "99.9% uptime" (do not write this without real data)
- Decorative "metric cards" composed of icon + number + word
- Mock tables filled with fancy fake data

If there is no real data, leave a placeholder or ask the user.

**❌ "Quote slop"**
Made-up customer testimonials and celebrity quotes used as page decoration. Leave a placeholder and ask the user for a real quote.

### Typography traps

**❌ Avoid these overused fonts**:
- Inter (the AI-generated webpage default)
- Roboto
- Arial / Helvetica
- Plain system font stack
- Fraunces (AI discovered it and ran it into the ground)
- Space Grotesk (AI's recent favorite)

**✅ Use a distinctive display + body pairing**. Directions:
- Serif display + sans-serif body (editorial feel)
- Mono display + sans body (technical feel)
- Heavy display + light body (contrast)
- Variable font for animated weight on hero titles

Font sources:
- Underused gems on Google Fonts (Instrument Serif, Cormorant, Bricolage Grotesque, JetBrains Mono)
- Open-source font sites (Fraunces siblings, Adobe Fonts)
- Do not invent font names from thin air

### Color traps

**❌ Inventing colors from scratch**
Do not design an entirely unfamiliar color system from zero. It usually does not harmonize.

**✅ Strategy**:
1. Has a brand color -> use it; interpolate the missing color tokens with oklch
2. No brand color but has a reference -> sample colors from the reference product screenshots
3. Completely from zero -> pick a known system (Radix Colors / Tailwind default palette / Anthropic brand); do not tune your own

**Defining colors with oklch** is the most modern approach:
```css
:root {
  --primary: oklch(0.65 0.18 25);      /* warm terracotta */
  --primary-light: oklch(0.85 0.08 25); /* same hue, lighter */
  --primary-dark: oklch(0.45 0.20 25);  /* same hue, darker */
}
```
oklch keeps the hue stable when you adjust lightness, which is more usable than hsl.

**❌ Casually inverting colors for dark mode**
Dark mode is not a simple invert. Good dark mode requires retuning saturation, contrast, and accent colors. If you are not going to do it properly, do not do it.

### Layout traps

**❌ Bento-grid overuse**
Every AI-generated landing page wants a bento grid. Unless your information structure genuinely fits bento, use another layout.

**❌ Big hero + 3-column features + testimonials + CTA**
This landing-page template has been beaten to death. If you want to innovate, actually innovate.

**❌ Identical cards in a card grid**
Asymmetric, varied sizes, some with images and some text-only, some spanning columns – this is what real designers do.

## Content rules

### 1. Don't add filler content

Every element must earn its place. White space is a design problem, solved with **composition** (contrast, rhythm, breathing room), **not** by stuffing content in.

**Filler-detection questions**:
- If you remove this content, does the design get worse? If the answer is "no", remove it.
- What real problem does this element solve? If the answer is "to make the page feel less empty", delete it.
- Does this stat / quote / feature have real data behind it? If not, do not invent it.

"One thousand no's for every yes."

### 2. Ask before adding material

You feel another paragraph / page / section would be better? Ask the user first; do not add unilaterally.

Why:
- The user knows their audience better than you
- Adding content has a cost; the user may not want it
- Adding content unilaterally violates the "junior designer reporting to a manager" relationship.

### 3. Create a system up front

After exploring design context, **state your intended system out loud first** so the user can confirm:

```markdown
My design system:
- Color: #1A1A1A body + #F0EEE6 background + #D97757 accent (from your brand)
- Type: Instrument Serif for display + Geist Sans for body
- Rhythm: section title with full-bleed colored background + white text; ordinary sections with white background
- Imagery: hero uses a full-bleed photo, feature sections use placeholders pending your assets
- At most 2 background colors to avoid visual noise

Confirm this direction and I will get started.
```

Wait for confirmation before working. This check-in avoids "discover halfway through that the direction was wrong".

## Scale specs

### Slides (1920x1080)

- Body text minimum **24px**, ideal 28-36px
- Title 60-120px
- Section title 80-160px
- Hero headline can use 180-240px display sizes
- Never use anything below 24px in slides

### Print documents

- Body minimum **10pt** (about 13.3px), ideal 11-12pt
- Title 18-36pt
- Caption 8-9pt

### Web and mobile

- Body minimum **14px** (16px for older audiences)
- Mobile body **16px** (avoids iOS auto-zoom)
- Hit target (clickable element) minimum **44x44px**
- Line height 1.5-1.7 (1.7-1.8 for Chinese)

### Contrast

- Body vs background **at least 4.5:1** (WCAG AA)
- Large text vs background **at least 3:1**
- Check with Chrome DevTools accessibility tooling

## CSS power tools

**Advanced CSS features** are a designer's best friends – use them generously:

### Typography

```css
/* Make heading line-breaks more natural so the last line is not a single orphan word */
h1, h2, h3 { text-wrap: balance; }

/* Body wrap that avoids widows and orphans */
p { text-wrap: pretty; }

/* CJK typography power tools: punctuation squeeze, line-start/end control */
p { 
  text-spacing-trim: space-all;
  hanging-punctuation: first;
}
```

### Layout

```css
/* CSS Grid + named areas = readability through the roof */
.layout {
  display: grid;
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
  grid-template-columns: 240px 1fr;
  grid-template-rows: auto 1fr auto;
}

/* Subgrid aligns content inside cards */
.card { display: grid; grid-template-rows: subgrid; }
```

### Visual effects

```css
/* Designer-flavored scrollbar */
* { scrollbar-width: thin; scrollbar-color: #666 transparent; }

/* Glassmorphism (use sparingly) */
.glass {
  backdrop-filter: blur(20px) saturate(150%);
  background: color-mix(in oklch, white 70%, transparent);
}

/* View Transitions API for silky page changes */
@view-transition { navigation: auto; }
```

### Interaction

```css
/* :has() makes conditional styling easy */
.card:has(img) { padding-top: 0; } /* cards with images have no top padding */

/* container queries make components actually responsive */
@container (min-width: 500px) { ... }

/* the new color-mix function */
.button:hover {
  background: color-mix(in oklch, var(--primary) 85%, black);
}
```

## Decision shortcut: when you are unsure

- Want to add a gradient? -> probably do not
- Want to add an emoji? -> do not
- Want to add rounded corners + a border-left accent to a card? -> do not, use a different approach
- Want to draw a hero illustration in SVG? -> do not, use a placeholder
- Want to add a quote for decoration? -> ask the user for a real quote first
- Want to add a row of icon features? -> ask whether icons are needed; they probably are not
- Use Inter? -> pick something more distinctive
- Use a purple gradient? -> use a color scheme grounded in something

**When you feel "adding this would make it look better" – that is usually a sign of AI slop.** Ship the simplest version first, and only add when the user requests it.
