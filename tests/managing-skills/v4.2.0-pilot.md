### 12.6 — Find-vs-filter decoupled

**Status:** ✅ PASSES (with nuance)

design-review IS a reviewer skill, so 12.6 applies. Analysis of the output structure (Process step 3):

- **Keep**: 3-5 things working
- **Fix**: severity-tagged (critical / important / polish)
- **Quick Wins**: top 3 changes <5 min

The Fix list **enumerates all findings with severity tags** (each item gets a tag at find-time, but no item is filtered out). Quick Wins is **additive curation** on top of the full Fix list — it surfaces the highest-leverage 3, but mid-severity items still appear in Fix.

This is the right pattern: enumerate-and-tag, then additively curate. The opposite (the anti-pattern) would be: "Fix: list only critical issues; ignore polish-level items."

**Framework finding:**

> 12.6 detection cannot be pure regex. "Quick Wins: top 3" looks like filter language to a regex but is actually additive curation. ms-reviewer must use semantic check (read the surrounding context) to distinguish *additive curation* (acceptable) from *exclusionary filtering* (anti-pattern).

**Adjustment for Phase 4:** ms-reviewer's 12.6 detection logic must read the section context, not pattern-match keywords. Specifically: if a "top N" phrase appears AFTER a complete enumeration of findings (additive), it passes; if it appears INSTEAD OF complete enumeration (exclusionary), it fails.

## Findings to fold into Phases 1-4

### Finding 1 (Phase 1): Section 12 needs N/A handling

For items 12.4 and 12.5, the validation rule should be: **N/A counts as a passing answer, not as a 0-score.** Wording for `pre-release-checklist.md`:

```markdown
**N/A handling:**

- 12.4 (fan-out): if the skill is single-threaded by design and has no parallel-eligible step, mark N/A. Pass.
- 12.5 (per-subagent overrides): if the skill does not delegate to subagents (no Agents table), mark N/A. Pass.

Score impact: N/A items contribute 0 to numerator AND 0 to denominator (i.e. removed from the scoring set, not zero-scored).

Effective Section 12 max ranges: focused skill (no fan-out, no agents) → 6.0; orchestrator skill (full applicability) → 8.0.

Pass threshold: 95% of applicable items, rounded down. So a focused skill needs ≥5.7/6.0; an orchestrator needs ≥7.6/8.0 (which we round to 8/8 in practice for blocking decisions).
```

**Plan revision impact:** total checklist max becomes a moving target (62 + applicable Section 12 items × weight). Pre-release-checklist needs to compute "applicable max" per-skill and threshold against that. Acceptable; same shape as Section 1 (already has CRITICAL all-must-pass logic).

### Finding 2 (Phase 4): 12.6 detection requires semantic check

ms-reviewer cannot grep for "top N" / "only" / "filter to" — these phrases appear in valid additive-curation contexts.

**Detection algorithm for Phase 4:**

1. Find every "top N" / "only the X" / "filter to" phrase in skill body.
2. For each match, read 3 lines before and after.
3. PASS if a complete enumeration appears in the context window (e.g. "Fix: severity-tagged list" precedes "Quick Wins: top 3").
4. FAIL if the phrase replaces enumeration (e.g. "Output: top 3 critical issues only").

ms-reviewer's existing P0-P3 framework supports this — emit "uncertain — manual review needed" for ambiguous cases. Don't auto-fail, surface for human judgment.

### Finding 4 (positive signal): patterns are well-formed for orchestrator targets

design-review is a focused skill, not an orchestrator. The cascade's hard work is on `managing-articles` (23 nested agents) and `managing-reports` (11 nested agents) — those are the orchestrator targets where 12.4 and 12.5 apply heavily. Pilot evidence supports the plan's "out of scope: design-* skills" decision: the design-* family won't move much under modernization because they're already in target shape.
