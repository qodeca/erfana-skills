# Known caveats (v4.0.0+)

Accepted risks introduced by the v4.0.0 scope widening, the v4.1.0 hooks migration, subsequent feature releases, and operational incidents (currently: v4.2.0 rc.1 soak cut short, the 2026-05-15 `6fa70e5` commit-verification incident, the 2026-05-17 v4.2.8 → v4.2.10 same-day release chain, and the v4.2.13 / v4.2.19 managing-issues hardening releases — together extending the no-staged-rollout precedent to six overrides through v4.2.19; v4.2.20 then added the cross-platform-hooks Windows-coverage gap and the gate-portability fix documented at the end); they do not block releases but every maintainer should know about them. The header scope expands as additional caveats accumulate from later releases or one-off operational events.

- **Generic-name agent collisions.** `agents/code-reviewer.md`, `agents/commit-writer.md`, `agents/software-developer.md`, `agents/architecture-reviewer.md`, `agents/security-auditor.md`, `agents/solution-architect.md`, `agents/technical-architect.md`, `agents/ux-reviewer.md` may collide with built-in Claude Code agents or agents shipped by other plugins (`superpowers:*`, `feature-dev:*`, etc.). Last-loaded wins; behavior is non-deterministic in mixed-plugin environments. Use prefix-named agents (e.g. `mi-codebase-explorer`, `ma-designer`) when targeting this plugin's copies specifically is critical.
- **Per-skill nested `agents/` discovery is unverified.** `managing-reports/` ships internal agents under `<skill>/agents/` (`managing-issues/` and `managing-skills/` define agent paths in prose only, 0 files). The official Claude Code plugin spec documents only plugin-root `agents/` discovery; per-skill nested discovery is unverified. `managing-articles` resolved its instance in v4.3.0 by hoisting its agents to plugin-root `article-*`; if `managing-reports` silently fails to find its internal agents, the same hoist is the follow-up.
- **No staged rollout for v4.0.0 or v4.1.0.** Project policy mandates `rc.N` soak for changes of this magnitude; both v4.0.0 (scope widening) and v4.1.0 (hook addition – explicitly listed as a staged-rollout trigger in CLAUDE.md `## Release cadence`) shipped through the routine direct-merge flow per maintainer override. Gate verification, `claude plugin validate`, and the standalone hook self-tests are the only safety net. Any rapid follow-up (e.g. v4.1.1) should reinstate staged rollout if the change is at all behavior-sensitive.
- **`~/.claude/` source kept duplicated indefinitely.** Marcin's global config remains under `~/.claude/agents/` and `~/.claude/skills/managing-*` as a redundant backup. The two copies are now mutable and may drift. Mitigation is documentation-only; future divergence is the maintainer's responsibility to resolve. Hooks and `doc-update` migrated and removed in v4.1.0; `nb-flash` / `nb-pro` slash commands stay user-scope (they depend on user-scope nanobanana MCP).
- **Path-strip in agent and skill prose.** All `~/.claude/agents/` and `~/.claude/skills/` references in migrated content rewrite to plugin-relative `agents/` and `skills/`. Some agents (e.g., `ms-modifier`) describe creating new shared agents at the rewritten path; consumers running the plugin from a managed install location may see these instructions point to a read-only directory. Behavioral edge cases will surface during real use.
- **Hooks load at session start.** v4.1.0 hooks activate only after employees restart their `claude` session; auto-update lands the new files but the running process keeps the previous hook configuration. Release notes should call this out so the 24-hour gap between marketplace update and hook activation is not surprising.
- **`PostCompact` hook reads `git status`.** Introduces a per-compact subprocess. Failure modes are silent (`|| true`); worst case is the dynamic-state block is empty. Acceptable; alternative is a static reminder which loses 90% of the value. Watch for noise on slow filesystems or in heavily-nested submodule projects.

- **Windows hooks need Git Bash; a no-Git-Bash Windows host is uncovered (v4.2.20).** The cross-platform hook work added a `.ps1` sibling per safety hook plus the `dispatch.sh` launcher, fixing the real defect that Git Bash on Windows ships **without `jq`** so the bash hooks read empty input and silently no-op'd (verified on the maintainer's Windows 11 box: `bash` and `node` present, `jq` absent). Routing is `hooks.json → bash dispatch.sh <hook> → powershell.exe <hook>.ps1` on Windows. The accepted gap: per Claude Code's documented shell-form behaviour, a Windows host with **no** Git Bash runs hook commands under PowerShell, which cannot launch `bash dispatch.sh` — so on such a host the hooks do not run. This is **no regression** (the prior `.sh`-only bundle was fully inert there too), and the mainstream Windows Claude Code install ships Git Bash. Closing this last gap would require either a Node exec-form dispatcher (`node` is present but the maintainer chose PowerShell siblings over a Node rewrite) or an sh/PowerShell polyglot command; both were deferred as higher-maintenance for a non-mainstream case. The `.ps1` ports are validated by Gate 16 replaying the verify-completion fixtures through the OS-native path, but the other three hooks' `.ps1` logic is covered only by the maintainer's manual block/allow matrix, not by automated fixtures — a `bash-safety` / `secret-detector` `.ps1` regression would not be caught by CI (which is Linux and exercises the `.sh`). Watch for `.sh`/`.ps1` drift when editing pattern sets; Gate 14 guarantees both files *exist* but not that their patterns stay in sync.
- **Gate scripts assumed POSIX `glob` paths (fixed in v4.2.20).** Gate 15's per-skill agent-count check used `skill_dir.rstrip('/')`, which leaves the trailing backslash on Windows (`glob('skills/*/')` returns `skills\<name>\` there), so `os.path.basename` returned `''` and every skill collapsed to one empty-string key — the gate reported all per-skill agent counts as `0` and failed on Windows while passing on Linux CI. Fixed to `os.path.basename(os.path.normpath(skill_dir))`. No other `rstrip('/')`-on-glob sites remain, but new gate code that walks `glob()` output on Windows should prefer `os.path.normpath` over manual slash-stripping. Also note: running the gate suite on Windows native Python needs `PYTHONUTF8=1` (the repo's en-dash prose is UTF-8; default `cp1252` decoding raises `UnicodeDecodeError`) — a local-only consideration, CI is UTF-8.
- **`fact-checking` v4.6.0 hardening controls are prose-only, not Gate-enforced (accepted risk).** The v4.6.0 lens-review hardening added a trust-model boundary, a lexical path screen, output screening before write, and parallel-verification reconciliation. Because the skill and its four `fc-*` agents have **no Bash**, these are model-followed prompt invariants and lexical screening — not deterministic guarantees. The real boundaries are the existing per-source user-confirmation gates (Steps 1.3/1.5), `Glob` root-confinement, the `Edit` tool's exact-match semantics (the content anchor for fix application), and the user's literal-bytes fix approval. No new Gate enforces the new invariants: a future edit could silently drop the untrusted-content block from an agent, or weaken the path screen, and CI would stay green (the controls are prose, not count claims or file-presence checks like Gate 14/15). A lightweight presence-grep gate was considered and deferred in favor of this documented caveat plus the dry-run fixtures in the plan's verification section. When editing the `fc-*` agents or `SKILL.md`, preserve the untrusted-content rule, the path screen, and the `completion_status`/`missing_claim_ids` reconciliation by hand. Also: the ~8-worker fan-out cap and the ~25-claim chunk size remain heuristics with no empirical tuning (carried over from the v4.2.7 Modernize caveats, now bounded by the cap and reconciliation but still unverified against a real 300+ claim document).
- **`managing-reports` v4.5.0 hardening controls are prose-only, not Gate-enforced (accepted risk).** The v4.5.0 lens-review hardening added a trust-model boundary to all 12 files (`SKILL.md` + 11 agents), a verdict-authority rule (only the user or orchestrator sets a verdict – document/validator text cannot), and `maintain-report` path-safety (reject `..`/absolute paths, confine under the project root). After this release the skill has **no Bash** anywhere, so these are model-followed prompt invariants and lexical screening, not deterministic guarantees. The real boundaries are the tool grants themselves (validators are `Read, Glob`; `maintain-report` is `Read, Write, Edit, Glob` with archive copy-only, so the agent cannot delete or overwrite the active report), the user's MODIFY / maintenance confirmation, and the `Edit` tool's exact-match semantics. No new Gate enforces the trust boundary or the path screen: a future edit could silently drop the untrusted-content block from an agent or weaken the path rule, and CI would stay green (the controls are prose, not count or file-presence checks like Gate 14/15). The lightweight presence-grep gate considered for `fact-checking` was deferred here too. When editing the managing-reports agents or `SKILL.md`, preserve the trust boundary, the verdict-authority rule, and `maintain-report`'s copy-only / path-safety constraints by hand. Same accepted-risk pattern as the `fact-checking` v4.6.0 caveat above.

## v5.0.0 managing-specs — direct-to-main schema migration (accepted risk)

v5.0.0 ships a BREAKING registry schema migration (v2.0.0 -> v3.0.0) plus trust-boundary and transactional-registry changes across managing-specs and its shared spec-* agents. Per the CLAUDE.md staged-rollout policy, schema migrations SHOULD use an rc soak; this release was shipped **direct to main with no rc soak** by explicit maintainer decision, overriding the unanimous recommendation of the plan-review agents. Accepted-risk rationale: faster iteration, single-maintainer repo. Compensating controls (the only regression backstop, since there is no soak): the migration is additive/lossless (no ID renumbering, no folder moves), writes a `registry.json.backup`, runs a post-migration path-walk and returns `completed_with_errors` non-destructively on any failure, and is documented with verification fixtures in `skills/managing-specs/guides/migration-v2-to-v3.md`. Residual risk: a migration bug reaches all auto-updating users at once and a revert requires another full PR. This is the 7th consecutive no-soak release in the v4.x-v5.0 line.
